# CS Seeder — what it is, how it works, how to use it

A short reference for the **CS Seeder** Salesforce app that powers the demo data on this droplet (and that Bell uses for Chatter writes).

> **Note on origin.** This doc was reconstructed from the seeder scripts and credentials on the droplet (`~/seed*.py`, `~/seed.env`) plus how the existing agents use it. The Salesforce-side setup details (exact app name in the UI, Run-As user) should be confirmed against Setup → External Client Apps when you next have it open.

---

## What it is

The CS Seeder is a **Salesforce External Client App** in your dev org with **admin-level write access**. It exists so two things can happen without a human clicking through Salesforce:

1. **Demo data can be created and reset** — accounts, opportunities, contracts, cases, tasks, contacts, all with realistic CSM signals (health bands, save plans, last-activity dates, etc.). Used to set up the renewal-risk pipeline demo.
2. **Bell can post Chatter notes** to Salesforce Opportunities automatically — the internal recap after a CSM-customer meeting. Chatter writes require admin/write scope, which is why Bell uses the Seeder's credentials rather than Tycho's read-only ones.

In short: **the Seeder is the one place we let an agent do admin writes in Salesforce, scoped tightly to internal records (data + Chatter), never customer-facing.**

---

## How it works in Salesforce

The Seeder authenticates via **OAuth 2.0 Client Credentials flow** — the same auth pattern Tycho uses, but pointed at a different app with different (write-capable) scope.

```
   ┌─────────────────┐                   ┌────────────────────────┐
   │  Seed script    │ ──── POST ─────▶  │ Salesforce OAuth       │
   │  or Bell's      │   oauth2/token    │ token endpoint         │
   │  sf_chatter.py  │   (client_creds   └────────────┬───────────┘
   │                 │    + key/secret)               │ access_token
   │                 │ ◀──────────────────────────────┘
   │                 │
   │                 │ ──── REST writes ────▶ /services/data/v60.0/sobjects/...
   │                 │      (Account, Opportunity, FeedItem, etc.)
   └─────────────────┘
```

Key facts:

- **No username / password / security token.** The org has SOAP login disabled. Auth is purely OAuth Client Credentials.
- **The app has a "Run As" user.** Every write the Seeder makes is recorded in Salesforce as having been performed by that user. Pick that Run-As deliberately — for the Seeder it should be a dedicated admin (not a real CSM, so demo data writes don't pollute a human's activity history).
- **It uses the standard Salesforce REST API** (`/services/data/v60.0/...`). Anything you can do through that API, the Seeder can do.

## What was needed to configure it

The setup boils down to four things — done once, then it just works:

### 1. The External Client App in Salesforce

Created in **Setup → External Client Apps**. Configured with:

- **OAuth flow:** Client Credentials enabled
- **Run As user:** an admin user
- **OAuth scopes:** at minimum `api` (full data access via API). Refresh token isn't needed for Client Credentials.
- **Consumer key + Consumer secret:** generated by Salesforce on app creation

### 2. The custom fields on Account / Opportunity

The seed scripts write to custom fields (`__c`) that have to exist on the org first. From `seed_pass1.py`, the Account needs:

| Field | Type | Purpose |
|---|---|---|
| `Segment__c` | Text/Picklist | SMB / MM / Ent — used by rubric |
| `ARR__c` | Currency | Annual recurring revenue |
| `Health_Score__c` | Picklist (Green/Yellow/Red) | Legacy field, restricted picklist |
| `Health_Band__c` | Text | Modern band field (`green`/`yellow`/`red`) |
| `Health_Score_Num__c` | Number 0–100 | Numeric health score |
| `Last_Login__c` | DateTime | For engagement signal |
| `Activated_At__c` | DateTime | For SOP 5-business-day-SLA check |
| `CSM_Save_Plan__c` | Long Text | The save plan note |

And on Opportunity:

| Field | Type | Purpose |
|---|---|---|
| `CSM_Forecast__c` | Picklist | "Positive Outlook" etc. |

Standard fields the seeder uses: `Account.Name`, `Account.Industry`, `Account.AnnualRevenue`, `Account.OwnerId`, `Account.Description`; `Opportunity.Name`, `StageName`, `CloseDate`, `Amount`, `Type`, `AccountId`; `Case.Subject`, `Status`, `Origin`; `Task.Subject`, `Status`, `ActivityDate`.

### 3. The Run-As user owning the right things

Two specific user Ids are baked into `seed_pass1.py`:

- `TAYLOR` (`005gK00004MQS1LQAX`) — the demo CSM who owns most accounts
- `MORGAN` (`005gK00004MPRA0QAP`) — second demo CSM
- `ADMIN`  (`005gK00004D7Vp7QAF`) — Angel's user, used for Compass Foods (the "missing CSM owner" SOP-5 case)

These are real Salesforce User Ids in the dev org. Recreating the seeder in a different org means swapping those for that org's user Ids.

### 4. The local credentials file

`~/seed.env` on the droplet:

```bash
SEEDER_INSTANCE_URL=https://<your-org>.develop.my.salesforce.com
SEEDER_CONSUMER_KEY=<from Salesforce External Client App>
SEEDER_CONSUMER_SECRET=<from Salesforce External Client App>
```

That's it — every script reads from that file.

## What else you can do with it

Because the Seeder has full API write access, it can do **anything in the org's data API**. Beyond what's already in the seed scripts:

- **Reset / wipe demo data** — write a "deseed" script that deletes accounts created by the Seeder (filter by owner = Taylor/Morgan or by a custom `Seeded__c` flag if you add one)
- **Add new signal types** — e.g. EmailMessage records for the meeting-transcript story, OpportunityContactRole links, more Cases for ticket-volume stories
- **Bulk update custom fields** — to A/B-test rubric variants (flip a batch of accounts to a new health band and see how Kepler scores it)
- **Post Chatter** to any record (this is what Bell does) — Account, Opportunity, Case, Contact, anything with a feed
- **Create test users** — e.g. add a third demo CSM if you want to test ownership-routing stories
- **Manage `Group` membership / queue assignments** — useful if you wire up case-routing or task-queue logic

A few things it can't or shouldn't do:

- **It can't send customer emails.** Salesforce email-send goes through a separate setting (Workflow / Email Alerts / `EmailMessage` with `SendEmailMessage`), and the safety pattern is to keep email-send out of the Seeder entirely. Bell sends from Gmail, not Salesforce, on purpose.
- **It shouldn't touch production data.** It's an admin app in the dev org. If a prod org ever exists, give it a separate External Client App with much narrower scope.
- **It shouldn't be Tycho.** Tycho is read-only on a different app with a read-only permission set. Don't conflate "Seeder writes admin-level" with "Tycho reads" — the read/write split is the safety model.

## Roberto's quick start

If you want to play with the Seeder:

### Read access to the credentials

The credentials live in `~/seed.env` on the droplet (`hermes@137.184.137.125`). Three values:

```bash
SEEDER_INSTANCE_URL    # your Salesforce My Domain
SEEDER_CONSUMER_KEY    # OAuth client id
SEEDER_CONSUMER_SECRET # OAuth client secret
```

### Smoke-test the connection

```bash
ssh hermes@137.184.137.125
python3 -c "
import requests
v={}
for line in open('/home/hermes/seed.env'):
    if '=' in line and not line.startswith('#'):
        k,val=line.strip().split('=',1); v[k]=val
inst=v['SEEDER_INSTANCE_URL'].rstrip('/')
r=requests.post(inst+'/services/oauth2/token',
    data={'grant_type':'client_credentials',
          'client_id':v['SEEDER_CONSUMER_KEY'],
          'client_secret':v['SEEDER_CONSUMER_SECRET']},timeout=30)
print('status:', r.status_code)
print('instance:', r.json().get('instance_url'))
"
```

A `200` and an `instance_url` mean you're authed.

### Run the existing seed (idempotent — safe to re-run)

```bash
cd ~
python3 seed_pass1.py   # 10 demo accounts + renewal opps
python3 seed_pass2.py   # cases, tasks, contacts (depends on pass1's id_map.json)
```

After `seed_pass1.py`, look at `~/runs/demo-seed/id_map.json` for the placeholder → real Salesforce Id mapping (handy if you want to grep for an account's Id later).

### Post a Chatter note to any record (the Bell pattern)

```bash
python3 /home/hermes/sf_chatter.py <SalesforceId> "Your Chatter body here"
```

That's literally how Bell does it. The script returns JSON with the `feed_item_id` on success.

### Write your own seed script

Use `seed_pyramid.py` as a template — it's the shortest. It shows:
- How to load `seed.env`
- How to authenticate (the same `connect()` function)
- How to use `simple-salesforce` to create Contracts, Opportunities, and update Accounts

The pattern is always: load env → get OAuth token → make REST or `simple-salesforce` calls → print JSON results so the script is greppable.

### Safety reminders

- **Don't run unfamiliar seed scripts against a Salesforce org you care about.** This is a dev org for a reason. If we ever point the Seeder at a real customer-data org, it needs its own narrower External Client App with restricted permission sets.
- **The Seeder doesn't send customer-facing email.** That gate lives in Bell + Gmail + the Operator Surface approval queue, deliberately separate.
- **Idempotency varies by script.** `seed_pass1.py` is safe to re-run. The per-account stories (`seed_pyramid.py` etc.) aren't — each run adds another Contract/Opp. Check before re-running.

## Where the Seeder fits in the platform

```
                                     ┌────────────────────────┐
   /home/hermes/seed_*.py    ────▶   │  CS Seeder             │
   /home/hermes/sf_chatter.py        │  Salesforce External   │  ──▶  Salesforce dev org
   (Bell's Chatter writes)           │  Client App (admin)    │       (Account, Opp, FeedItem, …)
                                     └────────────────────────┘

   ───────────────────────────────────────────────────────────────────────
   Read-only side (separate app, separate creds — Tycho):

   tycho -z "..."                    ┌────────────────────────┐
   /home/hermes/hermes-scaled-cs/    │  Tycho's app           │
     agents/tycho/bin/sf_reader.py   │  External Client App   │  ──▶  Salesforce dev org
                                     │  (READ-ONLY scope)     │       (SELECT-only)
                                     └────────────────────────┘
```

The Seeder is the write side of the demo. Tycho is the read side of the platform. They are deliberately different Salesforce apps with different scopes — a misbehaving agent on one side cannot reach the other.
